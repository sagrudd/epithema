# Epithema Code Structure and Module Naming Policy

Canonical governance context:
[Epithema Governance Manual](../epithema_governance_manual.md)

## Purpose

This policy establishes mandatory structural rules for the `epithema` and related codebases so that the project remains navigable, reviewable, and maintainable as a large volume of code is generated and revised over time.

The policy exists to ensure that code organization is deliberate rather than incidental, and that growth in the number of tools, modules, and integrations does not degrade the clarity of the codebase.

## Policy Statement

The Epithema project shall maintain a clear and durable internal structure.

To achieve that objective:

1. Rust module names shall, wherever practical, reflect the name of the method, tool, or capability they implement.
2. Code shall be grouped by functional domain rather than accumulated in large undifferentiated directories or files.
3. Functional grouping may follow bioinformatics method families, computation hierarchy, workflow layer, or another explicit architectural principle, but that principle must be applied consistently.
4. New code generation shall conform to the established module and directory structure rather than introducing ad hoc placement.
5. Structure preservation is a mandatory engineering concern and not an optional style preference.

## Normative Requirements

### 1. Method-Aligned Module Naming

Where a module implements a specific EMBOSS or Epithema method, the module name should match or closely reflect the method name.

Examples:

- `needle.rs` for Needle-style alignment functionality
- `water.rs` for Water-style alignment functionality
- `seqret.rs` for sequence retrieval/transformation functionality
- `complex.rs` for the `complex` tool

Where exact one-to-one naming is not practical, the module name shall still be recognizably derived from the implemented method or capability.

Avoid opaque or generic module names such as:

- `misc.rs`
- `helpers.rs`
- `stuff.rs`
- `impl2.rs`
- `new_logic.rs`

unless the module genuinely represents a narrowly defined shared utility layer and is placed within a clearly named higher-order package.

### 2. Functional Domain Grouping

Code shall be organized into explicit functional domains.

Acceptable grouping strategies include:

- bioinformatics method family
  - sequence handling
  - alignment
  - motifs
  - phylogeny
  - data retrieval
  - translation
  - restriction analysis
- computation hierarchy
  - core primitives
  - algorithms
  - service layer
  - CLI layer
  - bridge layer
- cross-cutting platform domains
  - IO
  - plotting contract
  - validation
  - documentation generation
  - R bridge

The project may use a hybrid of these strategies, but the grouping rules must remain legible and consistent.

### 3. No Flat Accumulation

The project shall not allow large numbers of unrelated tools or utilities to accumulate in a single flat directory or oversized file set.

If a directory begins to hold too many unrelated modules, it must be refactored into subdomains.

### 4. Shared Code Placement

Shared code must be placed according to purpose.

- Domain-specific shared logic belongs with its domain.
- Cross-domain abstractions belong in a clearly named shared crate or module.
- Utilities must not become a dumping ground for unrelated logic.

A utility module is acceptable only when its scope is explicit and narrow, for example:

- `fasta_utils.rs`
- `score_matrix.rs`
- `range_ops.rs`

A generic `utils` or `common` location must be tightly controlled and should contain only genuinely cross-cutting, low-level primitives.

### 5. Generated Code Conformance

Where code is generated by Codex or other tooling, the generated code must conform to the same structural rules as hand-written code.

Generated code shall not be accepted into the repository if it:

- bypasses functional grouping
- uses inconsistent module naming
- introduces duplicate structural patterns
- creates unexplained parallel hierarchies
- places method implementations in arbitrary locations

### 6. Architectural Consistency Over Time

Each newly introduced tool or capability must be placed into the existing architectural hierarchy.

If no appropriate location exists, the author must either:

1. create a clearly justified new domain grouping, or
2. refactor the current hierarchy so that the new capability has a coherent home.

### 7. Review Requirement

Code review must treat structure as a first-class acceptance criterion.

A change may be rejected on structural grounds even if the code is functionally correct.

Reviewers should evaluate:

- whether the module name matches the implemented method or capability
- whether the code has been placed in the correct domain
- whether shared logic has been extracted appropriately
- whether the change increases or reduces navigability
- whether the change introduces avoidable structural drift

## Recommended Directory Philosophy

The following is illustrative rather than mandatory, but it reflects the intended policy direction:

- `alignment/`
- `assembly/`
- `comparison/`
- `conversion/`
- `motif/`
- `phylogeny/`
- `restriction/`
- `retrieval/`
- `statistics/`
- `translation/`
- `io/`
- `plot_contract/`
- `service/`
- `cli/`
- `r_bridge/`
- `validation/`
- `docgen/`

Within such directories, method-specific modules should remain explicitly named.

Examples:

- `alignment/needle.rs`
- `alignment/water.rs`
- `retrieval/seqret.rs`
- `retrieval/refseqget.rs`
- `restriction/restrict.rs`
- `conversion/transeq.rs`

## Exceptions

Exceptions are permitted only where strict method-name alignment would reduce clarity.

Examples of acceptable exceptions include:

- a domain module that contains multiple closely related internal components
- a shared algorithm module used by several tools
- a bridge or adapter module named for the external system rather than a single method

Any exception should still maximize clarity and should be documented in code comments or surrounding module organization where needed.

## Enforcement

This policy applies to:

- hand-written Rust code
- Codex-generated code
- refactors
- new crates and modules
- R bridge-side Rust modules
- documentation-generation support code
- validation harness code where module structure is relevant

Maintainers should require conformance before merge.

## Governance Interpretation

This policy should be read together with:

- the architecture brief
- the tool-family policy
- the documentation and autodoc policy
- the Codex commit and push policy

Where there is tension between rapid code generation and structural clarity, structural clarity takes precedence.

## Closing Principle

Epithema is expected to grow into a large and durable codebase.

Its internal structure must therefore be treated as part of the product.

Clear module naming, domain grouping, and disciplined hierarchy are mandatory engineering requirements for the project and must be preserved as the codebase evolves.
